## Tasks

### Task 1 — docs/persistent-pubsub.md + docs/supervision.md

**Create `docs/persistent-pubsub.md`:**

Guide for Bayou's two publish/subscribe mechanisms. Structure:
1. **Two kinds of PubSub** — quick comparison table: `BayouPubSub` (in-memory, live-only, no
   replay) vs `BayouTopic` (log-backed, survives restarts, replayable). Rule of thumb: use
   `BayouPubSub` for ephemeral broadcast, `BayouTopic` when messages must survive restarts or
   late subscribers need history.
2. **`BayouPubSub` — in-memory broadcast** — `system.pubsub()`, `subscribe`, `publish`,
   `unsubscribe`; note dead actors are silently skipped; best-effort delivery. Code example.
3. **`BayouTopic` — persistent topic** — `system.topic(name, serializer)`, `publish`, `subscribe`;
   explain log-backed persistence. Code example: publish 3 strings, subscribe, receive them.
4. **Durable subscriptions** — `topic.subscribe(subscriptionId, ref)`: on reconnect, catch-up
   replay delivers everything missed since last delivery; `topic.unsubscribeDurable(id)` to forget.
   Code example: publish 5, subscribe durable, restart simulation (stop ref + spawn new ref with
   same subscriptionId), assert catches up.
5. **Replay from offset** — `topic.subscribeFrom(offset, ref)`, `topic.subscribeFromBeginning(ref)`,
   `topic.latestOffset()`; semantics: entries with seqnum > offset; pass `latestOffset()` to receive
   only future messages. Code example.
6. **Serialization** — `BayouSerializer<M>` interface; `JavaSerializer` for quick starts; note
   Kryo/Protobuf for production.
7. **Which to use?** — decision grid: ephemeral fan-out → `BayouPubSub`; durable delivery / late
   subscribers / audit log → `BayouTopic`.

Target length: ~170 lines.

**Create `docs/supervision.md`:**

Guide for building supervision trees. Structure:
1. **Why supervision?** — let-it-crash philosophy: actors crash on exceptions; supervisors decide
   what happens next. Code crashes bubble up to the supervisor, not to callers.
2. **Quick start** — `system.spawnSupervisor`, `SupervisorActor`, `children()`, `strategy()`;
   minimal runnable example (3-child supervisor with `OneForOneStrategy`).
3. **Supervision strategies** — comparison table: `OneForOneStrategy` (restart only crashed child,
   siblings unaffected) vs `AllForOneStrategy` (restart all on any crash). When to use each:
   independent workers → OneForOne; tightly coupled group → AllForOne.
4. **Restart window and death spiral guard** — `RestartWindow(maxRestarts, Duration)` limits;
   `RestartWindow.UNLIMITED`; exceeding the window triggers escalation to parent; top-level
   supervisor logs critical + stops. Code example showing window config.
5. **Custom strategy** — `SupervisionStrategy` is `@FunctionalInterface`; lambda example; decision
   table: `RESTART`, `RESTART_ALL`, `STOP`, `ESCALATE`.
6. **Dynamic children** — `sup.spawnChild(ChildSpec)` at runtime; use case: worker pools that
   grow on demand. Code example.
7. **Nested supervisors** — `ChildSpec.supervisor(id, supervisorActor)`; multi-level tree example;
   explain how escalation propagates up the tree.
8. **Decision tree** — ASCII tree or table: "One child crashed, siblings independent?" → OneForOne;
   "Children share resources (e.g. DB connection)?" → AllForOne; "Crash policy varies per child?" →
   custom strategy; "Group within larger system?" → nest in child supervisor.

Target length: ~200 lines.

### Task 2 — docs/patterns.md + README overhaul

**Create `docs/patterns.md`:**

Common actor patterns. Structure:
1. **Request-Reply** — `ref.ask(msg)` + `ctx.reply(value)`; `CompletableFuture`; timeout with
   `.get(1, SECONDS)`; note: only one reply is delivered (first `ctx.reply` wins). Code example.
2. **Fan-Out** — one actor broadcasts to N workers; two variants: (a) direct tell loop,
   (b) `BayouPubSub` topic. Code example for each.
3. **Work Queue (round-robin routing)** — router actor with a worker pool; dispatches messages in
   round-robin order. Code example: 3 workers, router keeps an index.
4. **Circuit Breaker** — state machine actor with CLOSED/OPEN/HALF_OPEN states using
   `StateMachineActor`; CLOSED passes through, OPEN rejects immediately, HALF_OPEN allows one probe.
   Code example using `StateMachineActor<State, Cmd>`.
5. **Pipeline** — chain of actors where each stage processes and forwards to the next;
   immutable message types flow left-to-right; code example: 3-stage pipeline (parse → validate →
   persist).
6. **Saga / Compensating Actions** — long-running process actor coordinates steps; on failure,
   sends compensation messages to already-completed steps; brief pattern description + skeleton.

Target length: ~200 lines.
