# Phase 14, Plan 1: Message Replay

## Objective

Add replay-from-offset to `BayouTopic<M>`. `subscribeFrom(long offset, ref)` delivers all log
entries with seqnum > offset, then transitions to live delivery — in-order, no gaps. 
`subscribeFromBeginning(ref)` is the convenience alias (offset = 0). `latestOffset()` returns the
current end-of-log seqnum synchronously. This requires a minor refactor: move `LogView` creation
from `TopicActor.preStart` into `BayouSystem.topic()` so `BayouTopic` can hold a direct reference
and expose `latestOffset()` without going through the actor mailbox.

## Context

**Current state:**
- `BayouSystem.topic()` creates `new TopicActor<>(k, serializer)` and `new BayouTopic<>(ref)` — neither holds a `LogView`.
- `TopicActor.preStart` initializes `logView = ctx.system().sharedLog().getView(LogTag.of("bayou.topic", name))`.
- `BayouTopic` has no access to `logView` and no `latestOffset()` method.
- gumbo 0.2.0: `logView.readAfter(long seqnum)` → `CompletableFuture<List<LogEntry>>` (entries with seqnum **strictly greater than** the given value); `logView.getLatestSeqnum()` → `long` (synchronous).

**Offset semantics (important):**
- `subscribeFrom(long offset, ref)` — internally calls `logView.readAfter(offset)`, giving entries with seqnum > offset. Callers pass offset = last-seen seqnum, or 0 to start from the very beginning (equivalent to `subscribeFromBeginning`).
- `subscribeFromBeginning(ref)` — alias for `subscribeFrom(0, ref)`. Works correctly assuming log entries have seqnum ≥ 1 (standard for gumbo). `readAfter(0)` returns all entries.
- `latestOffset()` — returns `logView.getLatestSeqnum()`. Returns 0 for an empty log; first entry has seqnum ≥ 1.

**Order guarantee:** `TopicActor` processes all messages on a single virtual thread. When `SubscribeFrom` is processed: (1) all prior `Publish` commands have already been appended to the log, (2) replay entries are delivered synchronously, (3) subscriber is added to `liveSubscribers`, (4) only then does the next queued `Publish` run. No gaps possible.

**Key design decisions:**
- **Refactor `LogView` creation into `BayouSystem.topic()`:** call `sharedLog.getView(LogTag.of("bayou.topic", k))` there, pass the `LogView` instance to both `TopicActor(name, serializer, logView)` and `BayouTopic(ref, logView)`. Remove `TopicActor.preStart()` (no longer needed).
- **`TopicCommand.SubscribeFrom<M>(long offset, Ref<M> subscriber)`** — new package-private record.
- **`TopicActor.handleSubscribeFrom`:** calls `logView.readAfter(offset).join()`, delivers to subscriber in-order (ignoring exceptions per message), then adds subscriber to `liveSubscribers`.
- **`BayouTopic.subscribeFrom(long offset, Ref<M> subscriber)`** — public API; tells `SubscribeFrom`.
- **`BayouTopic.subscribeFromBeginning(Ref<M> subscriber)`** — calls `subscribeFrom(0, subscriber)`.
- **`BayouTopic.latestOffset()`** — returns `logView.getLatestSeqnum()` directly (no actor message).
- `SubscribeFrom` subscribers become plain live subscribers after replay — no position tracking, no durable state.

Add handler:
```java
private void handleSubscribeFrom(long offset, Ref<M> subscriber,
                                  BayouContext<TopicCommand<M>> ctx) {
    try {
        List<LogEntry> entries = logView.readAfter(offset).join();
        for (LogEntry entry : entries) {
            try {
                subscriber.tell(serializer.deserialize(entry.dataUnsafe()));
            } catch (Exception ignored) {}
        }
    } catch (Exception e) {
        ctx.logger().error("Topic '{}': failed to replay from offset {}", name, offset, e);
    }
    liveSubscribers.add(subscriber);
}
```

Add import: `import com.cajunsystems.gumbo.api.LogView;`

Add three public methods:
```java
/**
 * Subscribe and replay all log entries with seqnum greater than {@code offset}, then receive
 * future messages live. Replay completes before any new publish is delivered to this subscriber.
 *
 * <p>Pass {@code offset = 0} (or use {@link #subscribeFromBeginning}) to receive all history.
 * Pass {@code offset = latestOffset()} to receive only future messages.
 */
public void subscribeFrom(long offset, Ref<M> subscriber) {
    actorRef.tell(new TopicCommand.SubscribeFrom<>(offset, subscriber));
}

/**
 * Subscribe from the very beginning of the topic log, receiving all historical entries
 * followed by live messages. Equivalent to {@code subscribeFrom(0, subscriber)}.
 */
public void subscribeFromBeginning(Ref<M> subscriber) {
    subscribeFrom(0, subscriber);
}

/**
 * Returns the seqnum of the most recently appended entry, or {@code 0} if the topic is empty.
 * Use as the {@code offset} argument to {@link #subscribeFrom} to receive only future messages.
 */
public long latestOffset() {
    return logView.getLatestSeqnum();
}
```

**Step 4: Update `src/main/java/com/cajunsystems/bayou/BayouSystem.java`**

```java
public <M> BayouTopic<M> topic(String name, BayouSerializer<M> serializer) {
    return (BayouTopic<M>) topics.computeIfAbsent(name, k -> {
        LogView logView = sharedLog.getView(LogTag.of("bayou.topic", k));
        Ref<TopicCommand<M>> ref = spawn("bayou-topic-" + k, new TopicActor<>(k, serializer, logView));
        return new BayouTopic<>(ref, logView);
    });
}
```

### Task 2 — MessageReplayTest

Create `src/test/java/com/cajunsystems/bayou/MessageReplayTest.java` with these test cases:

**Test 1: `doneWhen_subscribeFromBeginningReceivesAllHistoryThenLive`**

The exact done-when scenario:
- Publish 100 messages (strings "msg-1" through "msg-100")
- Wait for `topic.latestOffset() >= 100` (Awaitility, ensures all 100 are in the log)
- Spawn a subscriber, call `topic.subscribeFromBeginning(ref)`
- Wait for subscriber to receive 100 messages
- Publish 1 more ("msg-101")
- Wait for subscriber to receive 101 total
- Assert messages arrived in order: msg-1 ... msg-101

**Test 2: `subscribeFromOffsetReceivesOnlyMessagesFromThatOffset`**

- Publish 10 messages
- Wait for `latestOffset() >= 10`
- Record the seqnum at message 5 (use `latestOffset()` after publishing 5, before publishing 5 more) OR:
  - Simpler: publish 10, wait for `latestOffset() >= 10`, then `subscribeFrom(5, ref)` → should receive messages 6-10 (seqnum > 5)
- Wait for subscriber to receive exactly 5 messages
- Assert subscriber did not receive messages 1-5

**Test 3: `replayArrivesBeforeSubsequentLiveMessages`**

- Publish 5 messages
- Wait for `latestOffset() >= 5`
- Call `topic.subscribeFrom(0, ref)`
- Publish 5 more messages ("live-1" through "live-5")
- Wait for subscriber to receive 10 messages total
- Assert all 10 arrived in order: the 5 historical ones first, then the 5 live ones

**Test 4: `latestOffsetReflectsPublishedMessageCount`**

- Assert `latestOffset() == 0` before any publish (or handle that it may be > 0 on a shared log; use a unique topic name)
- Publish 5 messages, wait for each
- Assert `latestOffset() >= 5`
- Publish 5 more
- Assert `latestOffset() >= 10`

**Test 5: `subscribeFromDoesNotInterfereWithLiveSubscribers`**

- Spawn liveRef and call `topic.subscribe(liveRef)` (plain live subscriber)
- Publish 5 messages, wait for liveRef to receive 5
- Spawn replayRef and call `topic.subscribeFromBeginning(replayRef)` — gets all 5 replayed + any future
- Publish 5 more messages
- Wait for liveRef to receive 10 total, replayRef to receive 10 total
- Assert: liveRef received all 10 as live; replayRef received 5 via replay + 5 live = 10 total

For all tests: use unique topic names per test to avoid cross-test state (e.g., include the test method name in the topic name). Use Awaitility with a reasonable timeout (2-5 seconds).
