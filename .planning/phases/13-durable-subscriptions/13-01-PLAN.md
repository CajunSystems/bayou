# Phase 13, Plan 1: Durable Subscriptions

## Objective

Named subscribers track their read position in the topic log. When a durable subscriber goes
offline (actor stops/crashes), its last-processed offset is persisted via `logView.setValue()`.
When it re-subscribes with the same `subscriptionId`, `TopicActor` replays all missed entries via
`logView.readAfter(storedSeqnum)` before resuming live delivery. `topic.unsubscribeDurable(id)`
removes the subscriber AND deletes the stored position — subsequent re-subscribe starts fresh.

## Context

**Current state:** `TopicActor<M>` handles `Publish`, `Subscribe`, `Unsubscribe`. All subscribers
are live-only (no position tracking). gumbo 0.2.0 provides `logView.setValue(String, byte[])`,
`getValue(String)`, `deleteValue(String)`, `readAfter(long seqnum)`, and `getLatestSeqnum()`.

**Key design decisions:**

- **Two new `TopicCommand` records:**
  - `SubscribeDurable<M>(String subscriptionId, Ref<M> subscriber)` — durable subscribe
  - `UnsubscribeDurable<M>(String subscriptionId)` — cancel and forget position
- **`TopicActor` gets `HashMap<String, Ref<M>> durableSubscribers`** — safe as `HashMap` because
  all access is on the actor's single virtual thread (all commands go through the mailbox).
- **KV key format:** `"sub:" + subscriptionId` — stored on the topic's own `LogView`.
- **On `SubscribeDurable` (new subscription, KV is null):** record join point by calling
  `logView.setValue("sub:" + id, longToBytes(getLatestSeqnum())).join()`, then add to map.
  This marks "start from here"; if the subscriber goes offline immediately and re-subscribes,
  `readAfter(joinSeqnum)` correctly returns only messages published after the join point.
- **On `SubscribeDurable` (reconnect, KV is non-null):** call `logView.readAfter(storedSeqnum)`,
  deliver missed entries to subscriber, add to `durableSubscribers`. No KV update yet — the next
  live message will advance the position.
- **On `Publish`:** capture `AppendResult` to get `seqnum`. For each durable subscriber: if alive,
  `tell(payload)` then `setValue("sub:" + id, longToBytes(seqnum)).join()`. If dead: remove from
  map, do NOT advance position (so catch-up works on re-subscribe).
- **On `UnsubscribeDurable`:** remove from `durableSubscribers`, call
  `logView.deleteValue("sub:" + subscriptionId).join()`.
- **Append failure handling:** if `logView.append()` throws, log error, deliver to live subscribers
  only (best-effort — preserves Phase 12 behavior), skip durable delivery (can't track position
  without a seqnum). Phase 12 behavior for live subscribers is unchanged.
- **`longToBytes` / `bytesToLong`:** private static helpers using `ByteBuffer.allocate(Long.BYTES)`.
- **`BayouTopic` new methods:**
  - `subscribe(String subscriptionId, Ref<M> subscriber)` — tells `SubscribeDurable`
  - `unsubscribeDurable(String subscriptionId)` — tells `UnsubscribeDurable`

Add `handleSubscribeDurable`:
```java
private void handleSubscribeDurable(String subscriptionId, Ref<M> subscriber,
                                     BayouContext<TopicCommand<M>> ctx) {
    String key = "sub:" + subscriptionId;
    try {
        byte[] stored = logView.getValue(key).join();
        if (stored != null) {
            // Reconnect: replay missed entries
            long lastSeqnum = bytesToLong(stored);
            List<LogEntry> missed = logView.readAfter(lastSeqnum).join();
            for (LogEntry entry : missed) {
                try {
                    subscriber.tell(serializer.deserialize(entry.dataUnsafe()));
                } catch (Exception ignored) {}
            }
        } else {
            // New subscription: record join point
            logView.setValue(key, longToBytes(logView.getLatestSeqnum())).join();
        }
    } catch (Exception e) {
        ctx.logger().error("Topic '{}': error setting up durable subscriber '{}'", name, subscriptionId, e);
    }
    durableSubscribers.put(subscriptionId, subscriber);
}
```

Add `handleUnsubscribeDurable`:

private void handleUnsubscribeDurable(String subscriptionId, BayouContext<TopicCommand<M>> ctx) {
    durableSubscribers.remove(subscriptionId);
    try {
        logView.deleteValue("sub:" + subscriptionId).join();
    } catch (Exception e) {
        ctx.logger().error("Topic '{}': failed to delete position for '{}'", name, subscriptionId, e);
    }
}

Refactor `handlePublish` to capture `AppendResult` and drive durable delivery:
```java
private void handlePublish(M payload, BayouContext<TopicCommand<M>> ctx) {
    // 1. Persist to log
    long seqnum = -1;
    try {
        byte[] bytes = serializer.serialize(payload);
        AppendResult result = logView.append(bytes).join();
        seqnum = result.seqnum();
    } catch (Exception e) {
        ctx.logger().error("Topic '{}': failed to append to log", name, e);
        // Best-effort: deliver to live subscribers only
        deliverToLiveSubscribers(payload);
        return;
    }

    // 2. Deliver to live subscribers
    deliverToLiveSubscribers(payload);

    // 3. Deliver to durable subscribers + advance positions
    long appendedSeqnum = seqnum;
    Iterator<Map.Entry<String, Ref<M>>> it = durableSubscribers.entrySet().iterator();
    while (it.hasNext()) {
        Map.Entry<String, Ref<M>> entry = it.next();
        Ref<M> ref = entry.getValue();
        if (!ref.isAlive()) {
            it.remove(); // leave KV intact so catch-up works on re-subscribe
            continue;
        }
        try {
            ref.tell(payload);
            logView.setValue("sub:" + entry.getKey(), longToBytes(appendedSeqnum)).join();
        } catch (Exception e) {
            log.warn("Topic '{}': failed to deliver/advance for '{}'", name, entry.getKey(), e);
        }
    }
}
```

Extract `deliverToLiveSubscribers` from the existing Phase 12 delivery code:
```java
private void deliverToLiveSubscribers(M payload) {
    boolean hadDead = false;
    for (Ref<M> ref : liveSubscribers) {
        if (!ref.isAlive()) { hadDead = true; continue; }
        try { ref.tell(payload); } catch (Exception ignored) {}
    }
    if (hadDead) liveSubscribers.removeIf(ref -> !ref.isAlive());
}
```

/**
 * Register a durable subscriber. On reconnect (same {@code subscriptionId}), all messages
 * published since the last delivery are replayed before live delivery resumes.

public void subscribe(String subscriptionId, Ref<M> subscriber) {
    actorRef.tell(new TopicCommand.SubscribeDurable<>(subscriptionId, subscriber));

 * A subsequent {@link #subscribe(String, Ref)} with the same ID starts fresh from now.
 */
public void unsubscribeDurable(String subscriptionId) {
    actorRef.tell(new TopicCommand.UnsubscribeDurable<>(subscriptionId));
}

### Task 2 — DurableSubscriptionTest

**Create `src/test/java/com/cajunsystems/bayou/DurableSubscriptionTest.java`:**

Use same test setup pattern as `PersistentTopicTest` (SharedLogService + InMemoryPersistenceAdapter).

**Test 1: `doneWhen_durableReceivesAllNonDurableMissesGap`**

The exact done-when scenario from the roadmap:
- Spawn actorA (durable, id="sub-a") and actorB (non-durable)
- `topic.subscribe("sub-a", aRef)` and `topic.subscribe(bRef)`
- Publish messages 1–5 → wait for both to receive 5 (Awaitility)
- `topic.unsubscribe(bRef)`
- Publish messages 6–10 → wait for A to receive 5 more (total 10)
- `topic.subscribe(bRef)` (re-subscribe, non-durable)
- Publish nothing more
- Assert: A received 10 total, B received 5 total

**Test 2: `durableSubscriberCatchesUpOnResubscribe`**

Simulates subscriber going offline and catching up:
- Spawn actorA, `topic.subscribe("sub-a", aRef)` (new, no history)
- Publish 5 messages → wait for A to receive all 5
- Stop A: `aRef.stop().get(2, SECONDS)` → A is now dead
- Publish 5 more messages (A is dead — removed from durableSubscribers, position stays at msg5 seqnum)
- Spawn actorA2, call `topic.subscribe("sub-a", a2Ref)` → TopicActor reads stored position, calls readAfter, delivers msg6–10 to A2
- Wait for A2 to receive exactly 5 messages (Awaitility)

**Test 3: `unsubscribeDurableForgetPosition`**

Position is deleted on `unsubscribeDurable`; re-subscribe starts fresh:
- Subscribe A (durable, id="sub-a"), publish 5, wait for delivery
- `topic.unsubscribeDurable("sub-a")` — position deleted
- Publish 5 more
- Spawn A2, `topic.subscribe("sub-a", a2Ref)` — KV is null → new subscription, starts from now
- Publish 1 more trigger message → wait for A2 to receive exactly that 1 message
- Assert A2 did NOT receive the 5 missed messages (total = 1, not 6)

**Test 4: `twoSubscriptionsProgressIndependently`**

Two durable subscriptions on the same topic track positions independently:
- Subscribe actorA (id="sub-a") and actorB (id="sub-b")
- Publish 5 → both receive 5, positions both at seqnum5
- Stop A (goes offline)
- Publish 5 more → B receives 5 more (total 10), A removed from map, position stays at seqnum5
- Spawn A2, `topic.subscribe("sub-a", a2Ref)` → catches up with 5 missed messages
- Assert A2 received exactly 5, B received 10 total, independently
